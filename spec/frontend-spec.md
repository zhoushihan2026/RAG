+--------------------------------------------------+

## 3. 交互流程

### 3.1 问答流程

```
用户在底部输入框输入问题 → 点击发送或按 Enter
  → 输入框清空
  → 对话区显示用户消息气泡
  → 对话区显示助手消息气泡

  ===== 阶段一：思考阶段 (status 事件) =====
  → 显示动态思考动画:
     - 中心: 脉冲呼吸圆点 (3个渐变圆点，依次放大缩小)
     - 下方: 滚动文字带 (当前步骤文字从右向左滚动淡入)
     - 步骤示例:
       "正在识别公司名称..."
       "正在重写问题并分析问题类型..."
       "正在检索相关文档..."
       "正在生成回答..."

  ===== 阶段二：生成答案阶段 (stream_start + token 事件) =====
  → 思考动画平滑过渡为"正在生成"状态:
     - 呼吸圆点变为打字光标闪烁
     - 文字带固定为"正在生成回答..."
  → 【关键变更】token 事件到达时:
     - 后端可能发送两类 token:
       a) 推理过程 token (step_by_step_analysis 的 JSON 片段)
       b) 最终答案 token (final_answer 的文本片段)
     - 前端策略: 静默缓冲所有 token，不渲染到页面
     - 仅在 done 事件到达后，使用 details.final_answer 作为最终内容
     - [可选优化] 如果后端能区分推理 token 和答案 token，
       则仅流式渲染答案 token，跳过推理 token

  ===== 阶段三：完成阶段 (done 事件) =====
  → 思考/生成动画消失
  → 直接显示最终答案 (无中间 JSON 产物闪现)
  → 显示完整答案 + 推理过程(可折叠) + 参考来源
  → 自动滚动到最新消息
```

**v3 核心原则：用户永远看不到 LLM 的中间推理产物（JSON 字符串），只看到最终的格式化答案。**

### 3.2 文件上传流程

```
用户点击输入框左侧附件按钮 → 弹出文件选择对话框
  → 选择 PDF 文件
  → 输入框上方显示文件预览标签 (文件名 + x 删除)
  → 用户点击"添加到数据库" (自动上传) 或手动触发上传
  → 上传中: 文件标签显示加载状态
  → 成功: Toast 提示"上传成功"，文件标签消失
  → 已存在: Toast 提示"文件已存在"
  → 失败: Toast 提示错误信息
```

### 3.3 思考状态交互 (v3 重设计)

思考状态是核心体验，v3 从静态步骤列表升级为动态动画，让用户感知系统正在积极工作:

```
助手消息气泡出现时的完整生命周期:

  ┌─────────────────────────────────────────────┐
  │  阶段一: 思考中 (status 事件驱动)            │
  │                                             │
  │     ○  ○  ○   ← 3个脉冲呼吸圆点             │
  │    (渐变色: #2563EB → #8B5CF6 → #EC4899)    │
  │    (依次放大缩小，循环动画)                  │
  │                                             │
  │  ── 正在识别公司名称... ──                   │
  │     (文字从右向左滚动淡入，当前步骤高亮)      │
  │                                             │
  │  每个 status 事件:                           │
  │  - 上一步骤文字变暗并上移淡出                 │
  │  - 新步骤文字从右侧滑入高亮显示               │
  │  - 圆点持续脉冲呼吸                          │
  │                                             │
  ├─────────────────────────────────────────────┤
  │  阶段二: 生成中 (stream_start 事件触发)       │
  │                                             │
  │     |  ← 打字光标闪烁                        │
  │    (蓝色竖线, 2px宽, 18px高, 脉冲动画)        │
  │                                             │
  │  正在生成回答...                              │
  │     (固定文字, 无滚动, 配合光标闪烁)          │
  │                                             │
  │  【关键】: 此阶段 token 事件到达但            │
  │  不渲染任何文本到页面！                       │
  │  仅在内部缓冲区累积。                         │
  │                                             │
  ├─────────────────────────────────────────────┤
  │  阶段三: 完成 (done 事件触发)                │
  │                                             │
  │  动画消失，直接渲染最终答案:                  │
  │  - "最终答案" 标签 + 答案正文                │
  │  - 推理过程 (可折叠)                         │
  │  - 推理摘要 (可折叠)                         │
  │  - 参考来源标签                             │
  │                                             │
  │  【保证】: 用户从未看到中间 JSON 字符串       │
  └─────────────────────────────────────────────┘
```

#### v3 与 v2 思考状态对比

| 项目 | v2 (静态步骤) | v3 (动态动画) |
|------|--------------|---------------|
| 视觉形式 | 文字 + 旋转/勾号图标列表 | 呼吸圆点 + 滚动文字带 |
| 动画效果 | 图标旋转，静态切换 | 圆点脉冲呼吸，文字滚动过渡 |
| status 期间 | 逐步追加步骤行 | 步骤文字滚动替换 |
| stream_start 后 | 折叠为"思考完成"一行 | 过渡为光标 + "正在生成..." |
| token 期间 | **显示所有 token（含 JSON）** | **静默缓冲，不显示** |
| done 后 | 切换为 final_answer | 直接展示 final_answer |
| 用户感知 | 看到 JSON 闪现后变为答案 | 只看到答案，无中间产物 |

#### 状态判断逻辑

```typescript
// 组件内部状态判断
const phase = useMemo(() => {
  if (!thinkingSteps || thinkingSteps.length === 0) return 'idle';
  if (thinkingCollapsed) return 'generating'; // stream_start 后进入此状态
  return 'thinking'; // status 事件驱动中
}, [thinkingSteps, thinkingCollapsed]);
```

- `phase === 'thinking'`: 渲染脉冲圆点 + 滚动文字
- `phase === 'generating'`: 渲染打字光标 + 固定文字
- `phase === 'idle'`: 不渲染任何内容

### 4.7 StreamingText (流式输出) — v3.1 变更

**v3.1 行为: 最终答案通过打字机动画逐字展示。**

#### 后端限制说明

后端 SSE 的 `token` 事件推送的是 LLM 完整原始输出（包含 `step_by_step_analysis`、`reasoning_summary` 等字段的 JSON 字符串），**不是纯文本答案**。真正的答案文本仅在 `done` 事件的 `content.final_answer` 字段中提供。

因此，前端无法在 token 阶段实现真正的流式输出。采用 **"打字机模拟"方案**: token 阶段静默缓冲 + 生成动画，done 后用 TypewriterText 组件逐字展示 final_answer。

#### 事件处理流程

```
用户提问
  │
  ├─ status 事件 → ThinkingIndicator 显示思考动画（脉冲圆点 + 滚动文字）
  │
  ├─ stream_start 事件 → ThinkingIndicator 切换为生成动画（光标 + "正在生成回答..."）
  │                     前端进入"等待完成"状态
  │
  ├─ token 事件 (循环) → 静默缓冲，不渲染到 DOM
  │                     （内容为 LLM JSON 中间产物，不可展示）
  │
  └─ done 事件         → 获取 details.final_answer
                         → TypewriterText 组件启动打字机动画，逐字展示最终答案
                         → 打字完成后展示推理过程/摘要/参考来源（一次性）
```

#### token 事件处理 — v3.1 正确做法

- **静默缓冲**: `token` 事件的 content 在内存中累积，但 **不更新 message.content**
- **不渲染到 DOM**: 用户在此阶段只看到生成动画（光标 + "正在生成回答..."）
- **原因**: token 内容形如 `{"step_by_step_analysis": "1.问题要求...", ...}`，直接展示会造成图二的 JSON 闪现问题

#### TypewriterText 组件规格

```
组件: TypewriterText
位置: AssistantMessage 的最终答案区域
触发: done 事件后自动启动
输入: final_answer (完整字符串)
行为: 按固定间隔逐字符显示文字，末尾带闪烁光标
完成: 全部字符显示完毕后移除光标，触发 onDone 回调
参数:
  - text: string - 要展示的完整文本
  - speed: number - 每个字符的显示间隔(ms)，默认 15ms
  - onComplete: () => void - 打字完成回调
```

#### v3.1 三阶段展示策略

| 阶段 | 触发 | 内容 | 动画 |
|------|------|------|------|
| **阶段1: 思考中** | status 事件 | 脉冲圆点 + 滚动步骤文字 | 呼吸+滚动 |
| **阶段2: 生成等待** | stream_start + token 循环 | 左对齐光标 + "正在生成回答..." | 光标闪烁 |
| **阶段3: 打字机展示** | done 事件 | final_answer 逐字出现 | 打字机+光标 |
| **阶段4: 完成** | 打字机结束 | 推理过程/摘要/参考来源 | 无动画 |

#### 注意事项

- **推理过程、推理摘要、参考来源** 在打字机动画结束后才统一展示
- 这些内容不参与流式输出，是一次性整体显示的
- 如果 details 中无这些信息，则不显示对应区块
- **后续优化方向**: 若后端改造为仅推送 answer_token（不含中间产物），可改为真正的实时流式输出

## 5. 状态管理

### 5.1 全局状态

```typescript
interface AppState {
  // 消息列表
  messages: Message[];
  // 当前是否正在生成
  isGenerating: boolean;
  // 上传状态
  uploadState: 'idle' | 'uploading' | 'success' | 'error';
  // 已选待上传文件
  selectedFile: File | null;
  // 知识库文件列表
  kbFiles: KBFile[];
}

interface Message {
  id: string;
  role: 'user' | 'assistant';
  content: string;
  status?: string;
  isStreaming?: boolean;
  details?: MessageDetails;
  // v3 新增
  thinkingSteps?: ThinkingStep[];
  thinkingCollapsed?: boolean;  // stream_start 后设为 true，触发"生成中"动画
}

interface ThinkingStep {
  text: string;
  state: 'thinking' | 'done';
}

// v3 新增: 助手消息的显示阶段
type AssistantPhase = 'idle' | 'thinking' | 'generating' | 'done';
// - idle: 无思考步骤（初始态或错误后）
// - thinking: 有 status 事件驱动，显示脉冲圆点 + 滚动文字
// - generating: stream_start 到达，token 缓冲中，显示光标 + "正在生成..."
// - done: done 事件到达，显示最终答案
```

### 5.2 v1 行为

- 每次提问独立，不携带历史消息
- 无"新建对话"按钮 (v2 添加)
- 思考步骤随每条助手消息独立，不跨消息共享

### 6.1 问答请求 (v3 行为变更)

```typescript
async function sendQuestion(question: string, onEvent: (event: SSEEvent) => void): Promise<void> {
  // ... fetch 和 reader 初始化不变 ...

  // v3 关键变更: SSE 事件处理逻辑
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    // ... SSE 解析不变 ...

    switch (eventType) {
      case "status":
        // → 触发思考动画更新（脉冲圆点 + 滚动文字）
        onEvent({ type: "status", content: stepText });
        break;

      case "stream_start":
        // → 思考动画过渡为"生成中"状态（光标闪烁）
        onEvent({ type: "stream_start", content: "" });
        break;

      case "token":
        // === v3 核心变更 ===
        // 不再触发内容渲染！
        // 仅通知调用方有 token 到达（用于内部缓冲，可选）
        // 前端静默累积，不显示给用户
        // onEvent({ type: "token", content: tokenText });  // 可选：仍传递但 UI 不渲染
        break;

      case "done":
        // → 动画消失，直接使用 final_answer 渲染最终答案
        const details = JSON.parse(data);
        onEvent({ type: "done", content: details });
        break;

      case "error":
        onEvent({ type: "error", content: errorMsg });
        break;
    }
  }
}
```

**v3 SSE 处理策略总结:**

| 事件类型 | v2 行为 | v3 行为 |
|----------|---------|---------|
| `status` | 追加思考步骤行 | 更新滚动文字带 |
| `stream_start` | 折叠思考步骤，开始渲染 token | 过渡到"生成中"动画 |
| `token` | **实时渲染到页面** | **静默缓冲，不渲染** |
| `done` | 用 final_answer 替换内容 | 直接展示 final_answer |
| `error` | 显示错误文字 | 显示错误文字（不变） |
